# 赛博朋克卧室 — 项目复盘报告

## 项目概述

目标：为朋友制作一个赛博朋克风格的第一人称 3D 卧室网页，支持鼠标/触屏转动视角、WASD 移动、与房间物品互动，以及一个相册日历功能。

技术栈：Three.js (CDN) + 原生 JS + WebView2 桌面打包。

---

## 一、项目进程回顾

### 阶段 1：跳一跳游戏（顺利）

使用 Three.js 做了一个 3D 跳跳棋类游戏，经过几轮迭代改成了 QTE 蓄力机制，并打包成了 .NET WebView2 桌面应用。

**做对了什么：**
- 小步迭代，每次只改一个关注点
- 浏览器测试 → 桌面打包，流程清晰
- 遇到 WebView2 蓝屏时，通过日志定位到了 COM 线程模式和 SDK 版本不匹配

### 阶段 2：赛博卧室（失败）

从零构建第一人称 3D 房间，经历多轮修改后用户放弃。

**时间线：**
1. 骨架搭建 — 房间几何体 + 相机旋转 + 霓虹灯带 ✅
2. 场景建模 — 床、书桌、椅子、窗户、门、日历 ✅（但窗户黑屏）
3. 交互系统 — 悬停高亮 + 点击触发 ✅
4. 调整风格 — 去掉全息元素，改为现代赛博装修 ✅
5. 移动系统 — WASD + 碰撞检测 ✅
6. 美观调整 — 纹理、灯光、装饰 ✅
7. **反复修复窗口/日历/书桌** — 越修越糟 ❌

## 二、根本原因分析

### 1. 单文件架构失控

项目所有代码（HTML + CSS + JS）在一个文件中，最终膨胀到 1200+ 行。每次修改需要在大量代码中定位、替换，极易引入新的 bug。

**应该做的：**
- 拆分模块：`room.js`（场景）、`controls.js`（相机）、`interactions.js`（交互）、`calendar.js`（相册）
- 使用 Vite 或简单 HTTP 服务器解决 ES Module 的 file:// CORS 问题
- 每个模块可以独立测试和修改

### 2. 坐标系混乱是最大的技术难点

Three.js 中：
- `PlaneGeometry` 默认面朝 +Z
- `BoxGeometry` 六面材质统一
- 物体在 Group 内的局部坐标 vs 世界坐标
- 旋转后各轴对应关系改变

**具体踩坑：**

- **窗户黑屏**：PlaneGeometry 只有一面可见，放在墙后的玻璃从房间内看到的是背面。即使用了 `DoubleSide`，Canvas 纹理本身绘图太暗、emissiveIntensity 不够、雾衰减太强三个因素叠加导致始终看不到画面。
- **日历凸起**：Group 内放了 BoxGeometry 做边框 + PlaneGeometry 做屏幕，Box 有厚度导致突出墙面。
- **日历白屏**：重写时 Canvas 纹理坐标计算错误，文字画在了错误位置。
- **显示器两面发光**：把整个 Box 设为 emissive，6 个面都发光。应该用 Box 做机身（不发光）+ Plane 贴前面做屏幕（发光）。
- **书桌朝向**：deskGroup.rotation.y 旋转后，内部所有子物体的局部坐标轴跟着转了，导致键盘、鼠标位置全部错位。

**教训：** 3D 场景中，任何时候改变 Group 的旋转，都要重新验证所有子物体位置。最好在 Group 旋转为 0 时构建好所有子物体，最后再旋转整个 Group。

### 3. 没有可视化的调试手段

每次修改后只能靠用户描述"还是一样""黑屏"，我无法看到实际渲染效果。

**应该做的：**
- 在每个可交互物品旁放置临时的彩色小球（调试标记），确认位置正确后再移除
- 使用 `scene.add(new THREE.AxesHelper(1))` 查看坐标轴方向
- 截图对比工具
- 或者在代码中加入调试模式（按 F1 显示碰撞体线框、物体包围盒等）

### 4. 过度修改而非精准修复

当用户反馈"窗户黑屏"时，我反复改了：
- 窗户位置（从 -2.5 挪到 -2.2 再到 -2.46）→ 没解决根本问题
- 材质参数（emissiveIntensity 从 0.9 到 1.4 到 1.8 到 2.2）→ 治标不治本
- 墙面结构（完整面 → 三块拼 → 挖洞）→ 复杂化

但从未停下来做**最基础的排查**：
1. Canvas 纹理是否真的画上去了？（在页面上放一个 `<img>` 显示 cityCanvas.toDataURL()）
2. 玻璃 Plane 的面朝向是否正确？（用 AxesHelper 确认）
3. 雾的衰减距离是否刚好把窗户变黑了？

### 5. 需求理解偏差

用户多次表达的设计意图我理解有偏差：

| 用户说的 | 我理解的 | 实际意思 |
|----------|----------|----------|
| "赛博朋克装修" | 全息投影 + 漂浮屏幕 | 深色墙面 + 霓虹灯带，但家具是现代真实物品 |
| "电竞桌椅靠墙" | 侧面靠墙 | **背面**靠墙，人面对房间 |
| "粒子在墙上" | 墙面电路线（第一次） | 贴墙的微小光点（不是电路、不是空气浮尘） |
| "日历贴墙" | 薄盒子（2cm） | 几乎完全 flush（<5mm） |

**教训：** 在 3D 项目中，"靠墙""贴面"这类空间描述极其模糊。应该先用最简单几何体（Box + Sphere）确认位置，用户认可后再做精细模型。

## 三、技术难点总结

### 难点 1：Canvas → Three.js 纹理管线

```
Canvas 2D 绘制 → CanvasTexture → MeshStandardMaterial(map + emissiveMap) → PlaneGeometry
```

这个链条中每个环节都可能出问题：
- Canvas 绘制太暗（暗色背景 + 暗色建筑）
- `colorSpace = SRGBColorSpace` 忘加导致颜色错误
- `needsUpdate = true` 忘设导致切换内容不刷新
- `emissiveIntensity` 太低被环境光淹没
- 雾衰减让远处的纹理变黑

### 难点 2：第一人称控制的惯性系统

鼠标移动 → 速度累积 → 指数衰减：
```js
velocity *= Math.exp(-8 * dt);
```
衰减系数太大会导致视角"卡顿"，太小会导致"漂移"。8 是经验值，但不同帧率下表现不同。

### 难点 3：碰撞检测

用 AABB（轴对齐包围盒）做碰撞，但旋转后的物体（如书桌）其包围盒应该用 OBB（有向包围盒）。简化为 AABB 导致碰撞区比实际物体大很多，玩家被"空气墙"挡住。

### 难点 4：桌面打包的 COM 线程问题

WebView2 的 `EnsureCoreWebView2Async` 必须在 STA 线程调用。.NET 9 的 top-level statements 自动生成 MTA 入口点，导致 `RPC_E_CHANGED_MODE` 错误。需要用显式的 `[STAThread] static void Main()` 并确保 SDK 版本匹配系统运行时。

## 四、如果重来一次，我会怎么做

1. **先搭本地开发服务器**（`npx serve` 或 python `http.server`），解决 file:// 的 ES Module 限制
2. **拆分模块**：每个组件独立文件，修改不互相影响
3. **渐进构建**：先做纯色无纹理的灰盒房间 → 用户确认布局 → 加材质 → 加纹理 → 加交互 → 加细节
4. **每个 3D 组件先用调试标记验证**：在物体位置放彩色球、显示包围盒线框
5. **窗户问题单独排查**：在 HTML 中放一个 `<img>` 标签实时显示 cityCanvas 内容，确认 Canvas 绘制正确；然后单独创建一个测试页只渲染窗户
6. **更多向用户确认**：布局草稿、颜色方案先截图（或简单示意图）确认再写代码

## 五、对用户的建议

作为开发者，你在审阅 AI 生成的代码时有几点值得注意：

1. **3D 项目尤其容易积重难返**。如果一个组件连续两次修改都没修好，不要继续修——要求彻底重写那个组件，或者回退到工作版本。

2. **让 AI 解释它的修改方案再动手**。特别是在空间布局上，要求它用文字描述"桌子在哪个墙、面向哪个方向、坐标大致多少"，对齐理解后再写代码。

3. **要求阶段性验证**。"先把窗户纹理画在一个 HTML img 标签里让我看看"比"改完直接看 3D 场景"高效得多。

4. **善用 Plan Mode**。这个项目初期用了 Plan Mode 效果不错，但后期迭代时没有再用。每次大改前重新 Plan 能避免方向跑偏。
